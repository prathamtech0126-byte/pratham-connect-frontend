3 Ways to Add Payments to Reports

Which structure keeps existing code most intact?

A

Embedded section at bottom of Reports page Recommended

Add a <PaymentsSection /> component that renders below the existing sales/counsellor sections — only for superadmin. The component has its own filter state and API call, completely independent.

Reports.tsx — only 3 lines added at the bottom:
{/* Payments List — superadmin only */}
{user?.role === 'superadmin' && (
  <PaymentsSection />
)}
New file: components/payments/PaymentsSection.tsx
Has own filter bar (Today/Monthly/Yearly/Custom) + table

Pros

  • Minimal touch to Reports.tsx (3 lines)
  • Self-contained component
  • Appears naturally after sales content

Cons

  • Two filter bars on one page (but clearly separated)
B

Separate page — /reports/payments-list

A completely standalone page. Zero changes to Reports.tsx. A sidebar sub-link under Reports, visible to superadmin only, links to it.

Changes:
  • New file: pages/Reports/PaymentsListPage.tsx
  • Add route in App.tsx (1 line)
  • Add sidebar sub-link in Sidebar.tsx (superadmin only)
  • Reports.tsx: zero changes

Pros

  • Reports.tsx untouched
  • Clean URL, bookmarkable
  • No visual clutter on Reports page

Cons

  • Not literally "in" the Reports page
  • Sidebar.tsx needs a small edit
C

Top-level tabs: Sales | Payments

Replace the current period filter bar with a top-level "Sales Report / Payments List" switcher. Picking "Payments" swaps the entire content area.

Pros

  • Clean tab UX

Cons

  • Most invasive — restructures Reports.tsx significantly
  • Breaks the existing filter bar pattern
  • High risk of affecting existing code
Recommendation: Option A. Adds the Payments section naturally after the sales report with only 3 new lines in Reports.tsx. The component is fully isolated — its own state, its own API call. Existing behaviour is completely unaffected.
